# Unseen — клиент

Single-page application на Lit. Light DOM + глобальные CSS-файлы. Одна зависимость в рабочей сборке — `lit`. `@noble/hashes` через `shared/` — потоковый SHA-256 для файловых чанков.

## Архитектура

Клиент разбит на слои: маршрутизация и загрузка приложения, UI-компоненты (Lit), реактивное состояние (Signals), доменная логика (FSM сессии, send/receive, крипто-обёртки, Markdown, передача файлов), хранилище, транспорт, воркеры и глобальные lifecycle-хуки. Крипто-примитивы и wire-кодек — в общем workspace `shared/`. Дальше — детали по слоям.

## Routing

Разбор URL даёт один из трёх маршрутов:

- `landing` — корень `/`.
- `chat` — `/r402#<secret>`, где fragment содержит 256-битный секрет (base64url, 44 символа → 32 байта).
- `invalid` — всё остальное.

Приложение слушает `hashchange`, `popstate` и первичную загрузку:

- `landing` → монтируется `<landing-view>`.
- `chat` → монтируется `<chat-view>` с секретом.
- `invalid` → молчаливый редирект на `/` (`history: 'replace'`). Отдельной страницы для невалидного маршрута нет — это убирает различимое состояние на стороне клиента.

Умный remount: если корневой тег совпадает с целевым, элемент не пересоздаётся; для чата обновляется секрет, если он изменился.

| `peerMode`                            | режим peer (по RELAY `mode_upgraded`)                   |

## Session FSM — точки входа и инварианты

Оркестратор сессии при загрузке:

1. Разбор URL; невалидный маршрут → редирект (см. Routing).
2. Для чата читается запись `StoredSession` под непрозрачным ключом, выведенным из секрета.
3. Запись есть → `sessionMode = 'PRF'`, переход в `RESUMING` (unwrap session_key через WebAuthn PRF).
4. Записи нет → `sessionMode = 'RAM'` (по умолчанию), переход в `CONNECTING` (первый HELLO с intent `create`).
5. Claim Web Lock против повторной вкладки (`ifAvailable`); lock занят → мгновенный `terminate('duplicate_tab')`.

Любой сбой decrypt / counter gap / неверное направление в `ACTIVE` → `terminate(reason)`.

При финализации завершения: состояние становится `TERMINATED`; в PRF-режиме fire-and-forget отправляется сигнал, что per-room passkey можно удалить; запись `StoredSession` очищается; claim Web Lock освобождается; файловое состояние сбрасывается (см. «Reset file-state»).

Если причина не мгновенная (не `user_panic` / `duplicate_tab`) и состояние ∈ {ACTIVE, PEER_RECONNECTING, RECONNECTING} — переход в `FATAL_ENDING` на 3000 мс перед финализацией.

## Lifecycle-хуки

### Выгрузка страницы

Отслеживается `pagehide` / `beforeunload`. При выгрузке обработчик закрытия WS не завершает сессию принудительно: синхронная очистка хранилища во время unload стёрла бы PRF-зеркало и сломала бы переживание перезагрузки. На `pageshow` флаг сбрасывается.

### BFCache

При восстановлении из BFCache (`pageshow.persisted`) сессия принудительно завершается: состояние → `TERMINATED`, лента очищается, все непрозрачные OPFS-директории зачищаются. Сессия BFCache не переживает; страница начинает с нуля.

### Инвариант заголовка

`document.title` всегда равен `peer@unseen ~ $ wait` и переустанавливается на `visibilitychange` и `pageshow` (на случай, если расширения или devtools меняют заголовок). Никаких контекстных суффиксов и значка на favicon — заголовок в стиле терминала менее опознаваем в переключателе задач и на скриншоте ОС.

### View Transitions

Переходы состояний оборачиваются в View Transitions API, кроме списка исключений (`FATAL_ENDING`, `TERMINATED`) — анимация при завершении сессии неприятна. Если API недоступен — прямой вызов.

## Политика хранения

| Где                           | Что                                                    | Когда                                                                                                                             |
| ----------------------------- | ------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| `sessionStorage[storageKey]`  | `StoredSession` (см. `02-protocol.md §Storage schema`) | только PRF, запись на `UPGRADING_LOCAL → ACTIVE` и `REKEYING → ACTIVE`                                                            |
| `localStorage['c7XmK9-bN4q']` | `'en' \| 'ru'`                                         | языковая настройка                                                                                                                |
| Непрозрачная OPFS-директория  | файловые чанки                                         | живёт только во время передачи; очистка при TERMINATED / загрузке / BFCache                                                       |
| IndexedDB                     | —                                                      | не используется                                                                                                                   |
| Cookies                       | —                                                      | не используются                                                                                                                   |
| WebAuthn credential           | эфемерный passkey на комнату                           | живёт на authenticator пользователя (Touch ID / Windows Hello / iCloud Keychain); signal Unknown Credential при штатном terminate |

`storageKey` и `lockKey` — непрозрачные base64url из 11 символов, выведены через HKDF. Не содержат подстрок `unseen`, `session`, `room`, `lock`, `transfer`, `file`. OPFS dir name — то же правило.

## WebAuthn-потоки

### Opt-in upgrade (RAM → PRF)

В состоянии `UPGRADING_LOCAL` запрашивается регистрация passkey для комнаты. Параметры: платформенный authenticator с обязательной user verification, алгоритмы Ed25519 (предпочтительно) и ES256 (запасной), расширение PRF с eval на `prfSalt`, таймаут 60 с. Отображаемое имя косметическое (`Room 402 <случайные символы>`), не выводится из секрета или `roomId`; username и displayName совпадают, чтобы в менеджере паролей ОС не осело ничего лишнего.

Исходы:

- **Успех** → wrap session_key, запись `StoredSession`, `sessionMode = 'PRF'`, RELAY `mode_upgraded` → `ACTIVE`.
- **Отмена пользователем** → `ACTIVE` без изменений, остаёмся в RAM.
- **Сбой** (расширение PRF не выдало результат / транспортная ошибка) → `ACTIVE` без изменений, остаёмся в RAM с системным сообщением `mode_upgrade_failed`. Опциональный апгрейд: его провал не завершает сессию.

### Resume (PRF-зеркало найдено при загрузке)

Запрашивается аутентификация по сохранённому credential (обязательная user verification, расширение PRF на `prfSalt`, таймаут 60 с). Во время запроса — подсостояние `RESUME_LOCKED` (composer заблокирован). По результату PRF выводится `wrap_key`, разворачивается session_key, далее resume probe → `ACTIVE`.

Сбои (`passkey_resume_get_rejected`, `passkey_resume_no_credential`, `passkey_resume_prf_missing`, `passkey_resume_unwrap_failed`) → завершение сессии с соответствующей причиной; хранилище очищается при финализации.

### Signal на terminate

При штатном завершении менеджеру учётных данных (iCloud Keychain / Windows Hello и т.п.) отправляется необязательный сигнал, что per-room passkey можно удалить. Делается по возможности, сбои тихо игнорируются; если API недоступен — no-op.

## Защита от повторной вкладки через Web Locks

Эксклюзивный Web Lock под непрозрачным именем (`lockKey`, HKDF-derived) запрашивается в режиме `ifAvailable`: если lock уже занят другой вкладкой — `terminate('duplicate_tab')`. Успешный claim удерживается на всё время жизни сессии и явно освобождается при завершении; браузер дополнительно снимает lock на закрытии или падении вкладки. Поверхность угроз `lockKey` — та же, что у ключа `sessionStorage`.

## i18n

Язык (`en` | `ru`) — реактивный сигнал: определяется из `navigator.languages` с откатом на `en`. Перевод по ключу с подстановкой `{placeholder}`; для счётных форм — `Intl.PluralRules` (CLDR). EN — базовый словарь, RU — overlay той же формы; структурное равенство словарей проверяется типами на этапе компиляции, расхождение полей даёт ошибку typecheck.

`LANG_STORAGE_KEY = 'c7XmK9-bN4q'` — непрозрачный base64url со значением `en` | `ru`; единственный разрешённый ключ в `localStorage`.

Форматирование времени: `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`; источник времени — внедряемый интерфейс часов, в браузере на базе `new Date().toISOString()` (Temporal не требуется).

Имена эмодзи SAS (EN/RU) — отдельная карта поверх замороженного пула, не часть frozen-артефакта.

## Markdown subset

Поддерживаемое подмножество — inline: bold (`**`), italic (`*`), inline code (`` ` ``), link `[text](url)`; блочные: headings (`#`–`######` → bold), fenced code blocks (` ``` `).

Лимиты: до 500 инлайн-токенов (превышение → откат к простому тексту), глубина выделения до 3, парсинг не дольше 200 мс (превышение → воркер прерывается и перезапускается).

Парсинг выполняется в отдельном Web Worker. Воркер запускается заранее при монтировании `chat-view` и сообщает о готовности (`{ kind: 'ready' }`, как file-воркеры); `PARSE_TIMEOUT_MS` отсчитывается после готовности и ограничивает только время парсинга, не загрузку воркера. При превышении воркер прерывается и перезапускается, что ограничивает DoS через тяжёлый парсинг.

Безопасность ссылок: allowlist URL-схем `{http:, https:}`. Невалидный URL или другая схема → откат к буквальному `[text](url)`. Отображение: `text → host` всегда; host остаётся в punycode (не нормализуется).

Откат при bomb: предупреждение + простой текст, если parser возвращает > MAX_TOKENS или timeout.

## Санитизация Unicode

Санитизация Unicode применяется ко всем peer-сообщениям перед рендером:

- **Вырезание bidi-управляющих**: U+202A–U+202E (PUSH/POP), U+2066–U+2069 (ISOLATE), а также неявные метки LRM (U+200E), RLM (U+200F), ALM (U+061C) удаляются.
- **NFC normalize**: Unicode normalization для consistent rendering.
- **Ограничение Zalgo**: combining marks (категория Mn) ограничены 4 подряд.
- **Zero-width-символы**: сохраняются и обёрнуты в `<span class="zw-marker">` для подсветки через CSS (outline). Удаление сломало бы легитимные сценарии их использования, поэтому символы помечаются, а не вырезаются.

Отдельный строгий вариант для имён файлов (display и storage): NFC, удаление bidi / управляющих / разделителей пути, ограничение в 255 UTF-8 байт, ведущая точка для имён из одних точек.

## DOM message log

Скользящее окно ограничено 500 сообщениями в DOM. Входящие сообщения батчатся через `requestAnimationFrame` (минимум перерисовок); при переполнении старые вытесняются, и UI показывает индикатор «earlier messages removed (N)».

Отключение автопрокрутки: при прокрутке вверх лента не дёргается, появляется значок `chat.newMessages` с переходом к последнему сообщению.

Системный плейсхолдер `waiting_for_peer` снимается при подключении peer.

## SAS UX

`<sas-badge>` — non-modal, рендерится рядом с composer в ACTIVE. 5 эмодзи + локализованные имена в подписи. `aria-live="polite"` с задержкой 150 мс (чтобы избежать двойного объявления при быстрых сменах состояния). UI **не требует** клика-подтверждения — пользователи сверяют по второму каналу параллельно с активной сессией. Не сошлось — burn button.

## Burn UX

`<burn-button>` — two-step:

1. Первый click → armed state (~3 с).
2. Второй click в окне → `terminate('user_panic')` (INSTANT — skip FATAL_ENDING).

Безусловный escape hatch: срабатывает, даже если сессия не инициализирована или уже завершена, но UI не отреагировал. Синхронно очищается `sessionStorage`, затем выполняется `location.replace('/')`; редирект ждёт завершения очистки OPFS, но не дольше 500 мс.

## UI передачи файлов

### Composer attach

Кнопка-скрепка появляется в `<chat-view>`, когда `sessionState === 'ACTIVE'` и передача файлов поддерживается; до ACTIVE она скрыта (textarea остаётся доступной для ввода черновика). Активна кнопка только при полном условии:

```
fileTransferSupported && fileTransferReady && sessionState === 'ACTIVE' && transferActive === null && incomingActive === null
```

Иначе рендерится неактивной (идёт инициализация OPFS или уже активна передача). Единственный глобальный слот: одновременно одна передача в любом направлении.

Выбор файла → очистка имени → во вложение запоминаются файл и очищенное отображаемое имя. Превью-чип показывает очищенное имя, не сырое `file.name`. При отправке получателю уходит `file_offer`.

### Фазы file-bubble

`<file-bubble>` рендерит одно из шести состояний в зависимости от FSM Signals + AttachmentRecord:

| Фаза              | Render                                                                         | Кто видит                                                                      |
| ----------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------ |
| `offer`           | inline-flex системная строка: «← имя файла · размер [✓ принять] [✕ отклонить]» | получатель                                                                     |
| `awaiting-accept` | системная строка: «→ имя файла · размер [✕ отмена]»                            | отправитель                                                                    |
| `inflight`        | системная строка: «← имя файла · {percent}%» с отменой                         | получатель                                                                     |
| `verifying`       | системная строка: «✓ имя файла»                                                | отправитель (после готовности воркера, ждём подтверждения целостности от peer) |
| `attachment`      | user-message с именем файла как встроенная ссылка (скачивание по клику)        | обе стороны, после полного успеха                                              |
| `unavailable`     | системная строка: «имя файла · передача отменена или вытеснена»                | при отмене/вытеснении                                                          |

Фазы до вложения — inline-flex `chat__system` строки с минималистичными иконочными кнопками (область нажатия 44×44).

Attachment phase — user-message (направление in/out); имя файла — встроенная ссылка-кнопка, стилизованная под inline-ссылку. Клик запускает скачивание:

- источник OPFS: файл берётся из OPFS-хендла → сохранение через системный диалог или blob URL.
- источник sender: используется ссылка на File в памяти отправителя — после успешной передачи отправитель тоже видит свой bubble доступным для скачивания.

## File workers + FSM

### Проба возможностей и boot-sweep

Запускается один раз при загрузке чата. Перед boot-sweep вкладка захватывает shared Web Lock, имя которого совпадает с непрозрачным именем её OPFS-директории, — liveness-маркер живой сессии: захват строго до создания директории, браузер отпускает lock при уничтожении документа. Под exclusive lock `OPFS_LOCK_NAME` делает capability probe (OPFS root + `createSyncAccessHandle` + Web Locks + Transferable ReadableStream) + boot-sweep (удалить все директории с непрозрачными именами, кроме текущей сессии и директорий, чьи имена числятся held или pending в `navigator.locks.query()`). Timeout 3000 мс → `fileTransferReady = false`.

### Отправляющий воркер

Читает File потоково, режет на чанки и шифрует каждый (counter persist-AHEAD блоками по 64), отправляет с управлением потоком по кредитам (soft cap 192 KiB, hard cap 256 KiB — `BUFFER_THRESHOLD_BYTES`). По завершении считает потоковый SHA-256 и шлёт `file_complete { tid, sender_sha256 }`.

### Принимающий воркер

Под Web Lock пишет в OPFS через `SyncAccessHandle`. Каждый чанк расшифровывается и проходит проверку границ (seq монотонен, длина ≤ `CHUNK_DATA_MAX_BYTES`, суммарный объём ≤ заявленного), пишется в файл, обновляет потоковый SHA-256; периодически шлётся `file_progress`. На `file_complete` сверяется тройной инвариант (размер, число чанков, хеш); при совпадении — `file_complete_ack`, вложение регистрируется как `source: 'opfs'`.

### Timeouts

| Константа                  | Значение | Сторона                                                             |
| -------------------------- | -------- | ------------------------------------------------------------------- |
| `OFFERED_TIMEOUT_MS`       | 60 000   | отправитель отменяет offer, если accept не пришёл                   |
| `OFFER_PENDING_TIMEOUT_MS` | 60 000   | получатель отвергает offer, если пользователь не нажал accept       |
| `RECV_STALL_TIMEOUT_MS`    | 30 000   | получатель — быстрый отказ, если accept отправлен, но чанки не идут |

На любой timeout / cancel / fatal error — `cancelActive(reason)` оба-сторонний (RELAY `file_cancel { tid, side, reason }`).

### Политика файлов (download-only)

Файлы **download-only**: приложение не декодирует и не рендерит вложения inline — нет `createImageBitmap()` и никакого декодирования недоверенных байтов. Значит, нет риска native-decoder на контролируемых атакующим данных внутри приложения (декодирование возможно лишь когда пользователь сам открывает скачанный файл — вне области защиты). Любой тип, включая `image/svg+xml`, отдаётся только на скачивание.

### Reset file-state

При завершении сессии файловое состояние сбрасывается: активные передачи отменяются, blob-URL отзываются, OPFS-директория сессии удаляется здесь же (purge под exclusive lock `OPFS_LOCK_NAME`). Panic-редирект ждёт завершения purge, но не дольше 500 мс; страховка — sweep всех непрозрачных директорий при открытии лендинга, boot-sweep следующего чата и BFCache restore. Каждый sweep пропускает директории с живым liveness-lock, поэтому директория активной сессии в другой вкладке не удаляется; orphan-директории живой вкладки не имеют и удаляются. Purge собственной директории liveness-lock не учитывает. После panic-редиректа документ уничтожен и lock отпущен — лендинг-sweep зачищает директорию паниковавшей вкладки.

## Доступность

- WCAG 2.1 AA contrast (axe-core enforced как E2E gate).
- 44×44 — минимальная область нажатия для кнопок действий (встроенная ссылка с именем файла — исключение, WCAG 2.5.5/2.5.8 inline-link).
- `aria-live="polite"` для SAS badge, индикаторов статуса, системных событий.
- защиты `prefers-reduced-motion`: пропуск View Transitions + CSS `@media` отменяет `:active` scale transitions.
- Блокирующие подтверждения — inline-панель с `role="alertdialog"` (например, `RESUME_LOCKED`); деструктивные действия — двухшаговый `<burn-button>`.
- Клавиатурная навигация: textarea Enter / Shift+Enter, фокусные кольца кнопок через `:focus-visible`.

## SSG / SRI

На этапе сборки рендерятся два HTML-файла из шаблонов с плейсхолдерами из английского словаря: оболочка лендинга (`/`) и предрендеренный каркас чата (`/r402`).

Рабочая сборка:

- SHA-384 SRI для каждого ресурса → атрибуты `integrity="sha384-..." crossorigin="anonymous"` на `<script>` и `<link rel="stylesheet">`.
- `<link rel="modulepreload" integrity>` для общих чанков и воркеров — SRI на динамические импорты через preload.
- Разделение бандла (парсер Markdown и воркеры — отдельными чанками).
- Именование с хешем содержимого для сброса кеша.

Сборка для разработки: без SRI и без разделения бандла.
