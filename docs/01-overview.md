# Unseen — обзор

Unseen — веб-приложение для конфиденциального 1-на-1 разговора. Без регистрации, без аккаунтов, без хранения сообщений на сервере. Сервер видит только зашифрованные пакеты и непрозрачный `roomId`.

## Цели

- Сквозное шифрование (E2E) между двумя браузерами; relay-сервер криптоматериала не имеет.
- Взаимная аутентификация по второму каналу через SAS (5 эмодзи + локализованные имена).
- Эфемерность по умолчанию: вкладка закрылась — сессия мертва, никаких постоянных ключей.
- Восстановление сессии после F5 или обрыва сети — опционально, по явному согласию пользователя через WebAuthn PRF.
- Передача файлов до 100 MiB с per-chunk AEAD-целостностью.
- Двуязычность (EN/RU) из коробки.

## Не-цели

- Не Signal: одноразовые сессии, не повседневный мессенджер.
- Не PWA, не Service Worker, не offline first, не нативные мобильные приложения, не push.
- Не групповые чаты, не голос и видео, не федерация, не восстановление аккаунта.
- Не пост-квантовая криптография (нет нативной поддержки в Web Crypto).
- Не правдоподобное отрицание: подтверждённый SAS привязывает session key к двум устройствам.
- Не максимальная защита: 40-битный SAS уязвим к целенаправленному перебору в офлайне; нет маскирующего трафика; нет выравнивания длины (padding).

## Платформа

Только современные браузеры: **Chrome 149+, Safari 26.5+, Firefox 151+**. Никаких полифилов, никакого babel, никаких сборок под устаревшие браузеры. `Bun.build` с `target: "browser"` выдаёт чистый ESM.

| Возможность                                                         | Статус                                                                                |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `navigator.locks` (Web Locks)                                       | **обязательна** — защита от повторной вкладки. Без неё блокирующее предупреждение.    |
| `navigator.storage.getDirectory` (OPFS)                             | обязательна для передачи файлов; при отсутствии — UI файлов отключается.              |
| `PublicKeyCredential.isUserVerifyingPlatformAuthenticatorAvailable` | проверка возможности для opt-in PRF; при отсутствии — кнопка повышения защиты скрыта. |
| `Temporal`                                                          | **не** обязательна. Метка времени через `new Date().toISOString()`.                   |

Единственная зависимость клиента в рабочей сборке — `lit` (декларативный рендер). `@noble/hashes` — потоковый SHA-256 для целостности файловых чанков (Web Crypto `subtle.digest` не умеет потоковое хеширование).

## Доверенная доставка кода

Unseen защищает от скомпрометированного relay **только если клиентский код уже загружен честно**. Самостоятельное развёртывание: тот, кто его размещает, отвечает за целостность бандла перед своими пользователями. SRI встраивается в HTML на этапе сборки для защиты от подменённых ресурсов при честно загруженной HTML-обёртке.

## Архитектурный обзор

```
┌──────────────┐         ┌──────────────────┐         ┌──────────────┐
│   Client A   │◀─────▶│   Bun.serve      │◀─────▶│   Client B   │
│   (browser)  │   wss   │   relay server   │   wss   │   (browser)  │
└──────────────┘         └──────────────────┘         └──────────────┘
  знает:                   знает:                       знает:
  secret, handshake_key,   roomId,                      secret, handshake_key,
  session_key,             opaque blobs,                session_key,
  peer's eph_pub, SAS      counts, timings              peer's eph_pub, SAS
```

Два браузера открывают одну ссылку с 256-битным секретом в URL fragment. Из secret через HKDF выводятся `roomId` для сервера и секретные ключи (handshake, SAS-anchor, непрозрачные имена хранилища, PRF salt, OPFS dir). Клиенты подключаются по WebSocket, сервер сводит их в комнату с тем же `roomId` (максимум 2). Через зашифрованный handshake они обмениваются эфемерными открытыми ключами X25519 и выводят `session_key`. Из `session_key` выводится SAS — 5 эмодзи с локализованными именами, пользователи сверяют по второму каналу параллельно с активной сессией. Каждое сообщение шифруется AES-256-GCM с монотонным счётчиком и пересылается через сервер как непрозрачный блоб.

Все сессии стартуют в **RAM-режиме**: `session_key` extractable, но в storage ничего не пишется, любой WS close = TERMINATED. Пользователь может явно поднять защиту до **PRF-режима** через кнопку повышения: регистрируется WebAuthn passkey для комнаты, `session_key` оборачивается через AES-KW и пишется в sessionStorage, сессия переживает F5 и обрыв сети. Когда оба peer в PRF, происходит согласованный rekey через свежий X25519 ECDH в зашифрованном канале — новый `session_key` non-extractable, исходный extractable забывается. Это **двухступенчатая защита**: базовая RAM, повышение по явному выбору.

## Слои клиента

- **UI** — Lit-компоненты (light DOM, глобальный CSS): лендинг, экран чата, заголовок с действиями, рендер сообщений, SAS-бейдж, кнопка burn, переключатель языка, кнопка повышения защиты, индикатор режима, плитка передачи файла. Состояния peer-exit / reconnect / terminated и блокирующие подтверждения рисуются inline внутри экрана чата.
- **State** — реактивный слой: тонкие Signals поверх `EventTarget`. Хранят состояние FSM сессии, режимы свой и peer, роль, отсчёт попыток reconnect, ленту сообщений и состояние передачи файлов.
- **Domain** — бизнес-логика: машина состояний и жизненный цикл сессии, выделение счётчика и AEAD на отправке и приёме, X25519-handshake, WebAuthn-потоки (регистрация / resume / signal), проба возможностей PRF, claim Web Lock против повторной вкладки, парсер подмножества Markdown, санитизация Unicode, форматирование времени, конвейеры передачи файлов.
- **Storage** — одна непрозрачная запись в `sessionStorage` (схема `StoredSession`) и непрозрачная OPFS-директория сессии с boot-sweep и очисткой на BFCache.
- **Transport** — WebSocket (открытие / закрытие / отправка + бинарный кодек), преднабор backpressure с резервированием счётчика, экспоненциальный backoff reconnect с кумулятивным cap.
- **Workers** — вынос тяжёлых операций: парсинг Markdown, проба возможностей OPFS с boot-sweep, потоковое шифрование и расшифровка с верификацией файлов.
- **Lifecycle** — глобальные хуки: контроль выгрузки страницы, инвариант заголовка, обёртка View Transitions со списком исключений, принудительный terminate при BFCache-restore.
- **Crypto primitives** (общий workspace `shared/`) — X25519, HKDF, AES-GCM, AES-KW, rekey, потоковый SHA-256, деривация ключей комнаты и сессии, кодеки.

## Слои сервера

Один процесс `Bun.serve`, всё состояние в RAM:

- **Маршрутизация** — HTTP и WebSocket-upgrade на одном порту.
- **Wire** — маршрутизация HELLO по intent, транзакционная доставка ACK / PEER_JOINED, побайтовый форвард RELAY, Origin allowlist, извлечение клиентского IP.
- **Реестр комнат** — in-memory отображение `roomId → Room` с состояниями WAITING / PAIRED / HALF_OPEN; сборщик мусора по льготному периоду и таймауту ожидания.
- **Ограничение темпа** — per-IP token bucket на действие и per-connection bucket для RELAY.
- **Раздача статики** — SPA-оболочки и ассеты по allowlist, единый набор security-заголовков на любой ответ.
- **Keepalive** — периодический серверный PING держит NAT-маппинг тёплым.
- **Метрики** — опциональный экземпляр на приватном порту в формате Prometheus.
- **Логирование и конфигурация** — структурированный логгер с двойным allowlist, конфигурация через переменные окружения.

Никаких баз данных, никакого состояния вне RAM-процесса. Перезапуск = все комнаты теряются.

## Границы доверия

| Знание                                                   | Alice | Server                     | Bob  |
| -------------------------------------------------------- | ----- | -------------------------- | ---- |
| `secret` (URL fragment)                                  | ✅    | ❌                         | ✅   |
| `roomId` (HKDF(secret))                                  | ✅    | ✅                         | ✅   |
| `handshake_key`, `sas_anchor`, `prfSalt`                 | ✅    | ❌                         | ✅   |
| Эфемерные закрытые ключи X25519                          | свой  | ❌                         | свой |
| `session_key`, SAS                                       | ✅    | ❌                         | ✅   |
| Plaintext / file chunks                                  | ✅    | ❌                         | ✅   |
| Frame `type`, `kind`, `nonce`, `ct_len` (открыто в wire) | ✅    | ✅                         | ✅   |
| IP, размеры пакетов, тайминги                            | свой  | runtime-only (не логирует) | свой |

Сервер видит сетевой уровень и открытые служебные байты wire-протокола (тип фрейма, kind, nonce, ct_len). Не видит ни одного клиентского ключа и ни одного байта plaintext.

## Машина состояний сессии

13 состояний клиентского FSM. Переходы, доступные только в PRF-режиме, отмечены **(PRF)**.

```
                          ┌────────────────┐
                          │ IDLE           │ (landing, нет secret)
                          └────────┬───────┘
                                   │ открыли URL с #secret
                                   ▼
                          ┌────────────────┐
                          │ boot сессии:   │
                          │ stored mirror? │
                          └─────┬────┬─────┘
                                │    │
                       нет      │    │ да (PRF mirror найден)
                                ▼   ▼
                          ┌─────────┐ ┌───────────────┐
                          │CONNECT- │ │ RESUMING      │
                          │ ING     │ │ (PRF unwrap   │
                          └────┬────┘ │  на user      │
                               │      │  verification)│
                               │      └─────┬─────────┘
              ACK(initiator)   │      ACK + send
                               ▼     resume probe
                    ┌──────────────┐   │
                    │ WAITING_FOR_ │   │
                    │ PEER         │   │
                    └──────┬───────┘   │
                           │PEER_JOINED
                           ▼          │
                    ┌──────────────┐   │
                    │ HANDSHAKING  │◀─┘ (путь joiner пропускает WAITING)
                    └──────┬───────┘
                           │ decrypt OK; session_key extractable
                           ▼
                    ┌──────────────┐
            ┌─────▶│ ACTIVE       │◀──── resume_ack OK
            │       └──┬──┬──┬──┬──┘
            │          │  │  │  │
            │  user    │  │  │  │ peer dropped (PRF)
            │  click   │  │  │  ▼
            │  upgrade │  │  │  ┌──────────────────┐
            │          │  │  │  │ PEER_RECONNECTING│
            │          ▼ │  │  └────┬─────────────┘
            │ ┌──────────┐│  │       │ resume probe OK → ACTIVE
            │ │UPGRADING_││  │       │ grace expired → PEER_LEFT → FATAL_ENDING
            │ │LOCAL (PRF)│  │
            │ └─────┬────┘│  │  собственный WS dropped (PRF)
            │       │     │  ▼
            │       │     │  ┌──────────────┐
            └───────┘     │  │ RECONNECTING │  backoff [1,2,4,8,16,30,60]s
                          │  └──────┬───────┘  cap = 5 мин
            ┌──────────┐  │         │ timer fired → reopen WS
            │ REKEYING │  │         ▼
            │ (оба     │  │  ┌──────────────┐
            │ peer PRF)│  │  │ RESUMING     │
            └────┬─────┘  │  │ (после blip) │
                 │ оба    │  └──────────────┘
                 │ swapped│
                 │ done   │
                 └────────┘
                             ↓ любая фатальная причина
                          ┌──────────────────────────────┐
                          │ FATAL_ENDING (3 s overlay)   │
                          │ пропускается если reason ∈   │
                          │ {user_panic, duplicate_tab}  │
                          └──────────┬───────────────────┘
                                     │
                                     ▼
                          ┌──────────────────────────────┐
                          │ TERMINATED                   │
                          │ ключи стёрты из RAM          │
                          │ и sessionStorage,            │
                          │ Web Locks claim освобождён,  │
                          │ fire-and-forget signal       │

```

`RESUME_LOCKED` — короткое подсостояние RESUMING во время PRF unwrap (диалог Touch ID активен); composer заблокирован.

Переходы восстановления и rekey детализированы в `02-protocol.md` (§Восстановление сессии, §Двухступенчатая защита sessionStorage).

## Что выживает между перезагрузками вкладки

| Артефакт                 | RAM-режим                                      | PRF-режим                                                                  |
| ------------------------ | ---------------------------------------------- | -------------------------------------------------------------------------- |
| `session_key` после F5   | потерян (RAM-only, не персистится)             | восстанавливается через WebAuthn PRF + AES-KW unwrap                       |
| Обрыв WS                 | TERMINATED сразу                               | RECONNECTING (auto-backoff до 5 минут = серверный льготный период)         |
| Обрыв peer               | TERMINATED сразу                               | PEER_RECONNECTING (обратный отсчёт до серверного льготного периода)        |
| Закрытие вкладки         | TERMINATED (sessionStorage чистится браузером) | TERMINATED (mirror в sessionStorage умирает с вкладкой)                    |
| Panic-wipe (burn button) | мгновенный TERMINATED                          | мгновенный TERMINATED + `PublicKeyCredential.signalUnknownCredential(cid)` |
